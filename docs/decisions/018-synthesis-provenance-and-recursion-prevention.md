# Decision 018 — Synthesis provenance and recursion prevention

## Status
Adopted

## Date
2026-05-16

## Context

[[017-three-architectural-laws|Decision 017]] codified that interpretation is probabilistic
and non-authoritative. That law states the principle. This decision states the mechanism that
makes it enforceable.

The hidden failure mode: synthesized outputs become retrieval inputs. Operational briefs,
onboarding summaries, continuity reports — all are generated by a model from canon, but once
they exist as files or records, they are retrievable. If the next synthesis pass treats them
as canonical input, the system enters a feedback loop where:

- synthesized interpretation influences future synthesized interpretation,
- model hallucinations get reinforced across passes,
- "organizational mythology" emerges (a coherent-sounding narrative not grounded in canon),
- humans habituate to the synthesis and stop touching source canon.

This is not a hypothetical for systems like Prismo. It is the dominant failure mode for any
system that generates interpretive content over its own corpus. The cost to prevent it now —
before synthesis-service ships in Phase 3 — is days. The cost to retrofit after the corpus
is contaminated is months and may be unrecoverable.

## Decision

All synthesized content carries provenance metadata, lives in a separately-tagged retrieval
space, and is excluded from authoritative retrieval surfaces by default.

### Provenance metadata (required on every synthesis output)

```yaml
synthesis:
  generated_at: <ISO 8601>
  generated_by: <provider id, e.g. anthropic:claude-haiku-4-5-20251001>
  task: <capability name, e.g. operational-brief>
  source_ids: [<list of canon doc ids used as input>]
  source_hashes: [<content hashes of sources at synthesis time>]
  reviewed: false
  reviewed_at: null
  reviewed_by: null
```

Every synthesized record persisted anywhere — file, ChromaDB entry, SQLite row, API response
that gets cached — carries this block. No exceptions for "small" or "ephemeral" synthesis.

### Retrieval separation

ChromaDB metadata gains a top-level `record_type` field:

- `canonical` — sourced from canon files written by humans (or human-reviewed drafts).
- `synthesized` — generated by a Layer 3 runtime via a Layer 2 capability.
- `operational` — workflow-state-service records (sessions, workstreams, focus).

Retrieval surfaces filter on `record_type` explicitly:

- `active_doctrine` includes only `record_type=canonical`. Always.
- `recent_context` and `historical_context` include `canonical` by default; may include
  `synthesized` only when explicitly requested by caller.
- A new `expandable_context` channel may surface `synthesized` content with provenance
  visible, never authoritative.

### Exclusion from `active_doctrine` is permanent and load-bearing

Synthesized content can never enter `active_doctrine`. Period. If a synthesized observation
deserves doctrine status, the path is: synthesis → human review → human writes canonical
doc → canon → `active_doctrine`. The drafts-only constraint already implements this path.
This decision extends it to all synthesized output, not just Sukuna drafts.

### Return-to-source pressure

Synthesis outputs displayed to humans (OperationalBrief, onboarding, continuity reports)
MUST include:

- linked source ids (clickable in any UI; printable in CLI),
- a visible "synthesized" indicator,
- timestamp and generating runtime.

A synthesis output rendered without provenance is a Service Rule violation under
[[017-three-architectural-laws|Decision 017]].

### Acknowledgment of staleness

A synthesis output is stale when any of its source documents has changed since synthesis.
Stale synthesis is excluded from retrieval until regenerated or explicitly re-validated.
Source-hash comparison drives this; the doctrine-service is responsible for the check.

## Rationale

**Why mandatory provenance, not optional.** Optional provenance becomes absent provenance.
The cost of always carrying the metadata block is negligible (a few bytes); the cost of one
unprovenanced synthesis output sneaking into `active_doctrine` is potentially permanent
corruption of doctrine retrieval.

**Why a separate `record_type`, not a status field.** Status (`active`, `proposed`, etc.) is
a lifecycle concept on canonical content. Synthesized content has its own lifecycle (fresh,
stale, reviewed, archived) that is orthogonal. Conflating them would force semantically
broken queries.

**Why exclude synthesis from `active_doctrine` permanently.** The whole point of doctrine is
that it is the structural ground truth. Allowing synthesis in — even reviewed synthesis —
defeats the purpose. The human-review-then-canonicalize path costs a few minutes per
elevation and is what keeps the substrate trustworthy.

**Why source-hash staleness, not source-id staleness.** A source doc can be edited without
its id changing. Hash comparison catches actual content drift; id comparison only catches
deletion or rename.

## Consequences

- **synthesis-service in Phase 3 inherits this discipline from day one.** Its API must
  produce provenance blocks; its outputs must be tagged `record_type=synthesized`.
- **Existing `/brief` endpoint owes a backfill:** brief outputs currently lack provenance
  metadata. To be added before `/brief` is wired into any UI or scheduled job.
- **ChromaDB metadata schema gains `record_type`.** Existing records get `record_type=canonical`
  in the next re-index (`canon/` files are by definition canonical; `operational` records do
  not currently flow through ChromaDB).
- **Sukuna drafts** — currently markdown under `~/canon/<project>/docs/drafts/` — are
  borderline. They are synthesized but not indexed. Treat them as `record_type=synthesized`
  when they are indexed; until then, the drafts-only constraint
  ([[003-knowledge-base-source-of-truth|Decision 003]] in homelab — predecessor concept)
  already prevents them from leaking into active retrieval.
- **doctrine-service gains a responsibility:** monitor source-hash drift across synthesized
  records and mark stale ones.
- **The "organizational mythology" failure mode is structurally prevented**, not merely
  warned against in prose.
