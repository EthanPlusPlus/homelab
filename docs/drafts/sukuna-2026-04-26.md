## 3. Thinking

- **The warm-up protocol is a fiction.** Decision 012's "one question at a time, 2-4 exchanges" rarely happens — sessions open with a task and skip straight to work. Either enforce it mechanically (bootstrap reads N specific files unconditionally) or remove the fiction from CLAUDE.md and workflow.md. Aspirational instructions that aren't followed are worse than no instructions.

- **Canon has a write problem, not a read problem.** Every Phase 1, Phase 2, warm-up, and Sukuna run targets making Claude *read* canon correctly. But the real failure mode is: things that were decided don't get written, or get written stale. The Pattern 3 saga — three documents in three states after two decisions — is the symptom. A "what changed this session" gate at commit time would do more than any read-side enforcement.

- **The prismo VM_HOST bug is a canary.** A known 2-line fix has been documented for multiple sessions and not fixed. If Prismo's workflow (propose → approve → execute → canonise) was working, this would be gone. Its persistence suggests the workflow breaks down for small, low-stakes tasks — because the ceremony is disproportionate to the fix.

- **The Stop hook is friction that doesn't produce signal.** It fires on every response. Claude says "not applicable" most of the time. Sukuna running periodically surfaces the same issues the hook is meant to catch, without the per-response tax. If Sukuna cadence becomes regular, retire the Stop hook.

- **Mobile gateway (006) doesn't need Claude Dispatch.** The proposed-idea assumes the mobile entry point must be AI. But "trigger a task on Prismo from a phone" is solvable with a Telegram bot + SSH proxy. No new AI layer. The goal is a communication channel to the VM, not a second reasoning layer. Claude Dispatch solves a harder problem (autonomous execution) when a simpler problem (task input) exists.

- **Two VMs would obsolete most proposed-ideas.** The 8GB RAM constraint blocks Ollama, limits concurrent workloads, and underlies every "blocked on RAM upgrade" note in canon. A RAM upgrade to 32GB + a second VM (or increased VM allocation) is more enabling than the entire proposed-ideas list combined. The bottleneck is hardware; the proposed-ideas are workarounds.

- **Canon discipline enforcement is the wrong layer to optimise.** Prismo has spent multiple decisions on making Claude read canon before proposing. But the bottleneck was never "Claude didn't read the right docs" — it was "Claude skipped the MCP call and reasoned anyway." Inline Phase 2 (Decision 012) solved this adequately. Further investment in enforcement architecture has diminishing returns. Ship things instead.

- **Proposed-idea 002 (Gitea) will never have a trigger.** "Homelab demonstrates sustained stability" is undefined and unobservable. When is uptime sufficient? When is redundancy "some"? Without a concrete threshold (e.g. "30 days without a service restart" or "RAM upgrade complete + second machine accessible"), this idea ages out silently. Either define the trigger or close it.

- **Sukuna should run after every session, not on-demand.** The value degrades between runs as inconsistencies accumulate. The cost per run is low — one agent read pass. Schedule it as a cron or `Stop` hook replacement: run Sukuna once per day, push the draft, Ethan reviews async. This is the architecture proposed-idea 009 pointed toward without saying it.

- **The canon/projects separation (Decision 007) creates a subtle trap.** Working in `~/projects/<name>/` has no docs — MCP is the only path. But MCP being down silently (empty results, no error) causes reasoning from blank context with no indication anything is wrong. The fallback heuristic ("plausibly has prior canon → read ~/canon/") doesn't fire when MCP returns empty without error. A 1-line MCP health check at session bootstrap would catch this.
