### Structural findings

- **`~/canon/context-server/docs/STRUCTURE.md` has no `proposed-ideas/` section.** The homelab STRUCTURE.md and even STRUCTURE.md both define `proposed-ideas/` as a first-class folder with lifecycle rules. The context-server STRUCTURE.md omits it entirely, even though `~/canon/context-server/docs/proposed-ideas/` exists and contains three files. Any doc-structure query for context-server would not surface guidance on proposed-idea conventions.

- **`~/canon/flight-planner/docs/STRUCTURE.md` is a stub.** Only 16 lines, no folder descriptions, no naming conventions, no indexing boundaries. It defers to homelab STRUCTURE.md but does not document flight-planner-specific structure (e.g., `architecture/` exists in flight-planner). Compared to the even and homelab STRUCTURE.md files, which are comprehensive, this is significantly under-specified.

- **Frontmatter discipline is inconsistent across projects.** Homelab decisions 020–023 carry full YAML frontmatter (`id`, `title`, `status`, `record_type`, `date`). Homelab decisions 017–019 have no frontmatter — they use a `## Status` section heading only. Context-server decisions have a mix: 008 has frontmatter, 001 does not. Even decisions have no frontmatter. Flight-planner decisions have no frontmatter. This inconsistency means `doctrine-service`'s metadata-reconciliation capability (Decision 020 Day-1) will surface false positives across older decisions that predate the frontmatter convention.

- **`~/canon/homelab/docs/proposed-ideas/009-maid-canon-standardizer.md` status says `experimental` in notes (`notes: agent built, pending review cadence`) but the frontmatter `status` field is absent.** All other proposed-ideas in homelab use a frontmatter `status` field. This one uses only `notes` to convey state, making it invisible to any lifecycle filter that reads frontmatter.

- **`~/canon/homelab/docs/decisions/017-three-architectural-laws.md` has no frontmatter.** Decisions 020–023 have frontmatter with `id`, `title`, `status`, `record_type`, `date`. Decisions 017–019 do not. The Service Rule check (which validates HTTP routes) doesn't catch frontmatter gaps, but doctrine-service's metadata reconciliation will. The gap is worth flagging now before doctrine-service's first run produces a noisy report.
