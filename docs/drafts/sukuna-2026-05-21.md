## 2. Observations

- **Proposed-idea 006 (mobile gateway) is effectively superseded by Decision 021 but not marked as such.** Decision 021 establishes that all Layer 4 surfaces are thin ReviewItem renderers, and that CLI ships first, with WhatsApp and mobile as named subsequent surfaces. The architectural question "how do we get mobile access to Prismo?" has a clear answer in Decision 021. Proposed-idea 006 still frames this as an open question with no approach decided, which is misleading. It should be annotated as direction-resolved by Decision 021, with the remaining open question being build sequencing only.

- **`prismo review` CLI is listed as Phase 3 step 3 in v2-progress.md but has no corresponding proposed-idea or decision.** Steps 1 (ReviewItem endpoints) and 2 (synthesis-service) both became decisions (021 contains the spec). Step 3 is in the build queue with no decision record, no scoping doc, and no capability-contract entry. Given that `prismo review` is the first human-facing judgment surface — the thing that makes the entire synthesis loop useful — its design deserves at least a proposed-idea capturing the interaction model before it's built. The absence means it will be designed during implementation, which is how adapter ontology drift starts (Decision 022).

- **The observation week's empirical data has no canonical home.** Decision 021 mandates an observation week with specific metrics to track (emission rate, approval rate, rejection reasons, edit frequency, time-to-decision, duplicate rate). `prismo metrics` / `GET /workflow/metrics` ships those numbers. But where do the observation-week findings land? There's no proposed-idea or decision template for "observation findings → quality gate adjustment." If the observation week surfaces patterns that should tune the confidence threshold or dedup sensitivity, those tuning decisions would presumably become annotations on Decision 021 or new decisions. This isn't tracked anywhere; the observation-week process doesn't close to canon.

- **The `SynthesisProvider` interface defined in Decision 015 predates the two-tier model discipline in Decision 023, and the two decisions are not reconciled.** Decision 015 defines `synthesize(task, context, max_tokens) -> str` as the single contract. Decision 023 introduces a second tier (`/synthesis/analyze` with `ANALYSIS_MODEL`) operating at a different capability level with web search tool use. Decision 015's interface covers neither the tool-use requirement nor the multi-turn reasoning loop. As synthesis-service grows, the gap between the abstract `SynthesisProvider.synthesize()` contract and the richer capability required for interpretive augmentation will widen. There's no decision tracking this divergence or proposing how the abstraction evolves.

- **Proposed-idea 010 (interaction observer) is described as "proposed, not blocked, not yet scoped" but is actually superseded in substance by Decision 021.** The interaction observer's core function — watching sessions, extracting decisions, proposing canonization — is exactly what synthesis-service Class 1 (internal synthesis: captures + canon + sessions → ReviewItems) does. The proposed-idea also asks "does it write a draft to drafts/, like Sukuna, or surface candidates inline?" — this question is answered by Decision 021. The only residual question is the hook/transcript access mechanism (what does session log access look like in Claude Code?). The proposed-idea should be annotated to reflect that Decision 021 subsumes its purpose, with the open question narrowed to the session-log-access mechanism.

- **There is a gap between what the `prismo capture promote` deprecation means in practice and when it will be enforced.** Decision 021 deprecates `prismo capture promote` and says it "will be removed once synthesis-service is the sole capture consumer." Synthesis-service has shipped. The condition for removal is met, but no follow-through decision or proposed-idea tracks the removal. The command presumably still exists in `scripts/prismo`, accepting arguments and executing a deprecated flow. The deprecation has no deadline and no removal ticket.

- **The `prismo` CLI script's deprecation trajectory (v1-service-map.md: "deprecated — replaced by Layer 2 API calls in V2") has no action record.** The V1 service map marks `scripts/prismo` as deprecated and to be replaced by Layer 2 API calls. But the CLI has been actively extended throughout V2 (adding `prismo review`, `prismo prior-art`, `prismo synth`, `prismo metrics`). The CLI is Layer 4, not Layer 2 — the v1-service-map entry was written before the Decision 021 framing clarified that CLI is a valid, permanent Layer 4 surface. The `v1-service-map.md` entry is now misleading: `scripts/prismo` is not being deprecated; it is the primary Layer 4 surface until a web UI ships. The entry should be corrected before it creates confusion for a new contributor reading V2 orientation docs.

## 3. Thinking

- **ChromaDB is a custom-deployment liability that has a commodity replacement.** Prismo uses ChromaDB in embedded mode for semantic retrieval. ChromaDB requires its own container, its own persistence volume, and its own rebuild process. **pgvector** (PostgreSQL extension, MIT license) would collapse vector storage into the same Postgres instance Prismo could use for workflow-state-service's SQLite content — removing a container, simplifying backups, and gaining mature ACID guarantees. The `sentence-transformers` embedding layer stays unchanged; only the vector store swaps. Relevance: Decision 001 chose ChromaDB for simplicity, but SQLite + ChromaDB is now two persistence systems where one Postgres + pgvector could do both. Research target: pgvector vs ChromaDB retrieval quality parity, migration path from ChromaDB to pgvector, whether `sqlalchemy` + `pgvector` integrates cleanly with the existing FastAPI layer.

- **SQLite for workflow-state-service is approaching the limits of its appropriate scope.** Workflow-state-service persists sessions, workstreams, captures, review items, and stale acknowledgments in SQLite. This was the right call for a single-contributor system (Decision 014: "SQLite keeps runtime footprint small"). But the ReviewItem schema is now relational — ReviewItems reference captures, sessions, synthesis runs, and canon documents. As more contributors and more surface types arrive, multi-writer SQLite will become a bottleneck. **PostgreSQL** is the natural upgrade: same HTTP API, same ORM layer (SQLAlchemy already in use), but production-grade concurrency, foreign key constraints, JSONB for flexible fields, and no WAL limitations. Research target: whether the existing SQLAlchemy models need schema changes for Postgres, connection pooling with FastAPI + asyncpg, migration tooling (Alembic is the standard).

- **The synthesis-service prompt template is hand-rolled; there are mature OSS frameworks that handle exactly this.** Synthesis-service uses a custom prompt template in `synthesis/analyze.py` (and presumably the Class 1 runner). **LangChain** and **LlamaIndex** are well-known but heavyweight. More relevant: **DSPy** (Stanford, MIT license) compiles prompt programs — instead of hand-writing templates, you specify a signature and let DSPy optimize the prompt against examples. For a system where synthesis quality is the core value metric and rejection_reason is being empirically tracked, DSPy's ability to auto-optimize prompts from observed outputs is directly applicable. Research target: DSPy compatibility with Anthropic SDK + web_search tool use, whether DSPy's compilation loop conflicts with Prismo's human-judgment architecture, whether the quality gate tuning problem is better solved by DSPy or by manual calibration.

- **The web search tool use in `/synthesis/analyze` is Anthropic-native; it will break when the model tier changes.** `prismo prior-art` uses `web_search_20250305` — a tool type specific to the Anthropic API. If Decision 015's SynthesisProvider abstraction is applied consistently, interpretive augmentation should be provider-agnostic. But web search is not provider-agnostic today. **Tavily** (purpose-built web search API for AI, paid but cheap, REST-based) or **SearXNG** (self-hosted metasearch, AGPL) would decouple web search from the Anthropic provider contract. Research target: Tavily vs SearXNG for structured research retrieval quality, whether a `WebSearchProvider` abstraction (mirroring `SynthesisProvider`) is the right pattern, what the interface should look like to be provider-agnostic across Anthropic, OpenAI, and local model tiers.

- **`prismo review` TUI has no prior art analysis in the canon but the problem is well-solved in OSS.** The ReviewItem review flow (list / show / approve / reject / edit) is essentially a structured inbox. **Gum** (Charmbracelet, MIT) provides composable terminal UI primitives (confirm, choose, input, spin) that could implement `prismo review` as a clean TUI without the overhead of a full TUI framework like Textual. Decision 021 says CLI ships first — Gum is exactly the right tool for CLI-first interactive review surfaces. **Textual** (also Charmbracelet, MIT) is the right tool if a richer layout is needed (multi-pane, scrolling content preview). Research target: Gum vs Textual for the ReviewItem review flow, whether the review interaction is simple enough for Gum alone or needs Textual's layout primitives.

- **The doctrine-service relationship graph is being built from scratch; knowledge graph tooling already exists.** Decision 020 scopes doctrine-service to include `getSupersessionGraph` and `resolveRelationships` — deterministic graph traversal over canon relationships (decision supersession chains, upstream/downstream relationships). This is a property graph problem. **Neo4j** (AGPL for community edition) is the canonical graph DB but heavyweight. **kuzu** (MIT, embedded columnar graph DB) is a recent alternative that runs in-process like SQLite. For Prismo's scale (hundreds to low thousands of canon nodes), kuzu would provide a real graph query language (openCypher) without requiring a separate service. Research target: kuzu vs custom SQLite adjacency tables for supersession graph traversal, whether an embedded graph DB is justified at current scale or if simple SQLite adjacency is sufficient until the canon grows.

- **The entire annotation/frontmatter validation problem in doctrine-service is solved by existing linting tools.** `doctrine-service`'s metadata-reconciliation capability (Decision 020) will build rules for: missing lifecycle fields, invalid statuses, malformed frontmatter, duplicate IDs. **Markdownlint** (Node.js, MIT) handles markdown structure. **yamllint** handles YAML frontmatter syntax. More relevantly, **markprompt** and custom **remark** pipelines can enforce frontmatter schemas against a spec. The question is whether Prismo should build its own linting rules (Python, integrated with the indexer) or wrap existing CLI tools as a pre-indexing validation step. Research target: whether a Python-native frontmatter validator (e.g., `pydantic` schema validation against extracted frontmatter) is preferable to wrapping yamllint, and whether any OSS tool handles the full Prismo frontmatter schema (`status`, `record_type`, `superseded_by`, `id`) out of the box.
