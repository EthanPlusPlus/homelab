Nothing has been recorded about the actual app: no decisions, no architecture, no constraints, no progress. The repo at `github.com/EthanPlusPlus/even` exists but no canon describes it. This is the dominant fact about the canon and warrants a decision about whether to backfill or write canon as decisions are made going forward.

An agent following the workflow gets empty MCP results and has to fall back to direct reads — and there is nothing to read either.

Concept assumed, never defined within the even canon (it may be defined in homelab canon, but even does not link to it).

- `progress.md` "Next" section defers to `proposed-ideas/` — but with the folder absent, this is a dangling pattern: progress is governed by a folder that has never existed. - No decision record exists for the choice of iOS as the platform, for receipt-OCR strategy, for splitting model (per-item assignment vs. percentages vs. shares), or for backend/no-backend. These are the load-bearing decisions of a receipt-splitting app and the canon is silent on all of them. Largest gap.
- The empty-section placeholder phrasing drift (4 variants) is a pattern with no governing decision. Trivial, but symptomatic — STRUCTURE.md governs naming and folder semantics but not in-file conventions.

## 3. Thinking

- The even canon currently exists for its own sake. There is nothing to retrieve. Delete the placeholder context files and the canon shrinks to two files (CLAUDE.md, STRUCTURE.md) — and arguably nothing is lost. Stop pretending scaffolding is canon.
- Or: just delete the whole thing and write canon when the first real decision happens. A canon with zero decisions is a yak; the repo lives at github.com/EthanPlusPlus/even — write code first, canonize the choices that actually got made.
- The Prismo canon discipline (Decision 012) requires consulting canon before proposing. For even there is nothing to consult. The discipline silently no-ops, which trains the agent to skip the step even once content arrives. Risk: discipline rusts in the gap between "scaffolded" and "populated."
- Receipt-splitting is a solved-shape product (Splitwise, Tab, etc.). The interesting decisions are: (1) on-device OCR vs. cloud (Vision framework vs. Textract/Document AI), (2) per-item assignment UX, (3) whether to even have a backend. These should be the first three decision records. None exist. Write them before writing Swift.
- "Allocate who ate what" is in the tagline. That is one specific splitting model (per-line-item to one or more eaters). Other models (% split, shares, weighted) are not mentioned. Either commit to per-item-only as a product decision and document it, or broaden the tagline. Right now it is implicitly a decision-by-tagline.
- The `localhost:8000/index` instruction in CLAUDE.md is fragile — it assumes the agent is running on the indexer host. If that is intended (Prismo only), say so. If not, remove it. Currently it is a footgun for a remote agent.
- `progress.md` deferring "Next" to `proposed-ideas/` is over-engineered for a one-developer project. Just put Next directly in progress.md and skip the folder until you have ideas worth structuring with reasoning. STRUCTURE.md is mimicking homelab canon at a scale that even does not warrant.
- The STRUCTURE.md "Just enough structured, retrievable documentation — never more" guiding principle is being violated by STRUCTURE.md itself. It declares five folder types for a project with zero contents in any of them. The structure file is currently the largest doc; if removing it would not hurt, it should not exist (per its own rule). Half-serious: delete STRUCTURE.md until the first decision record needs a home.
- No mention of receipts being multi-currency, taxes, tips, service charges, or item modifiers (e.g., "split this beer 3 ways"). These are the real edge cases that kill receipt-splitting apps. They are absent from open-questions.md and constraints.md. Open-questions should not be empty for a real product.
- The canon-MCP loop (write canon, re-index, query MCP) is the workflow for homelab. For even, with no canon, the loop runs dry. Consider explicitly documenting that even uses homelab's MCP but does not yet contribute to it — or skip the MCP step from the workflow until there is content. CLAUDE.md is currently lying to the agent about utility.
- Rename: `CLAUDE.md` at canon root is for the agent. Move the agent instructions into a section of `STRUCTURE.md` (or a `README.md`) and let the canon root have one human-readable entry point. Two top-level files for an empty canon is one too many.
