## 2. Observations

- **The `scripts/sukuna.md` agent is now orphaned by its own supersession.** Proposed-idea 013
  (the documented path for Sukuna → ReviewItems) was marked `superseded` on 2026-05-31, with the
  claim that Decision 026 (Sukuna Facet) + Decision 028 (Response Processor auto-capture) solve it
  "structurally." But they don't fully: the Sukuna *Facet* loads context, and the Response Processor
  auto-captures from conversational model output — neither replaces a *scheduled, structured
  three-section pass that emits one ReviewItem per finding*. This very run proves the gap: it ran as
  a markdown-writing script, not via the Facet or the processor. The replacement mechanism for the
  drafts/ violation has been declared solved while the violating script still runs. This is a real
  contradiction between PI-013's superseded-claim and operational reality — not a wording nit.

- **"Observation week" closure has a provenance gap.** `v2-progress.md` Phase 3 step 4 says "No
  canon closure record exists yet" for observation week, yet Decision 027 (observation-week-closure)
  exists and recent-changes references it. The progress doc was not updated when 027 landed. Phase 3
  step 4 should point to Decision 027.

- **Portability (Decision 032) silently invalidates dozens of canon cross-references but no
  migration is tracked.** 032 names `ubuntu-server.tail58b10c.ts.net` as first-class debt to replace
  with `PRISMO_HOST`. That hostname is hardcoded in every project CLAUDE.md, mcp.md, system.md,
  services.md, multiple runbooks, and the prismo CLI. None of those are flagged as
  pending-portability-debt. When the env-var audit (032 build step 1) happens it will touch ~15 canon
  files; there is no checklist anchoring that work. A `proposed-idea` or progress entry enumerating
  the coupling surface (032 lists it in prose) would make the debt trackable rather than re-discovered.

- **Auth is now load-bearing but exists nowhere in canon as a build artifact.** Decision 032 makes
  auth the thing that replaces Tailscale as the security layer (build step 2), and Decision 031
  defers it as "second slice." `open-questions.md` still frames MCP auth as a vague "should the MCP
  require authentication for production use?" These three say different things about the same
  question. Auth has crossed from open-question to committed-constraint and deserves its own
  proposed-idea or decision, not a stale open-question bullet.

- **The four-layer model is assumed everywhere but the canonical definition is buried and drifting.**
  The masterplan defines Layers 1–4; Decision 026 redefines them as "governance tiers (ownership +
  authority), not request flow / pipeline stages"; PI-015 lists a slightly different gloss (Layer 2 =
  "operational/workflow state"). The architect facet README repeats yet another phrasing. There is no
  single canonical "what the layers mean" doc — the definition is reconstructed from scattered
  annotations. For a system whose central reasoning frame is the layer model, this is the single most
  important concept lacking a home. (See Section 3 — this is the strongest argument for a
  `architecture/model.md`.)

- **`prismo-ui` shipped before its component canon caught up, repeating a known pattern.** prismo-ui
  has a CLAUDE.md and STRUCTURE.md but zero decisions/architecture/runbooks despite a full Next.js app
  with 7 panels, CORS changes to context-server, and a deploy story. The same lag was explicitly
  flagged and paid down for context-server ("Phase 2.5 implementation backfill — decisions 010–014…
  Pays down component-canon decision-record debt user flagged"). prismo-ui is accruing the identical
  debt one day in. At minimum: a deploy runbook (the GitHub-push-pending note in recent-changes is
  operational state that will be lost) and a decision recording the Tailscale-gated-no-auth choice.

- **The masterplan and roadmap are increasingly annotation-encrusted rather than rewritten.** Both
  v2-masterplan.md and v2-roadmap.md now carry multiple inline `**Annotation YYYY-MM-DD**` blocks
  correcting their own body text (Sukuna v2 dropped, CLI-first inversion, Layer 3.5 realization,
  Initiative 3.1 complete). The documents have become palimpsests: the original prose says X, the
  annotation says "actually not X." This is honest but is reaching the point where a reader can't
  trust the body without scanning for contradicting annotations. These are Phase-0/1 founding docs
  describing a system that no longer matches them. (See Section 3 — candidate for a clean rewrite or
  demotion to `architecture/history/`.)
