  drafts/ violation has been declared solved while the violating script still runs. This is a real
  contradiction between PI-013's superseded-claim and operational reality — not a wording nit.

  canon closure record exists yet" for observation week, yet Decision 027 (observation-week-closure)
  exists and recent-changes references it. The progress doc was not updated when 027 landed. Phase 3

  migration is tracked.** 032 names `ubuntu-server.tail58b10c.ts.net` as first-class debt to replace

## 3. Thinking

### Topic 1 — Canon structure for V2

- **The single highest-leverage move: rename `homelab` → `prismo` now.** The forcing function
  Decision 032 and prismo-ui already arrived. The repo is the de facto system canon, every memory
  file apologizes for the name ("historical accident"), and STRUCTURE.md spends three paragraphs
  explaining the lie. Portability-for-commercial-viability means an outsider clones this — and the
  first thing they see is a repo called "homelab" that is actually the whole product. The deferral
  reasons are all now satisfied. Do it; leave a `homelab/` subfolder for the literal infrastructure.

- **Split the conflated repo into `prismo-system/` (governance) and `homelab-infra/`
  (hardware/Proxmox/VM/networking).** STRUCTURE.md admits the repo "carries two things at once." Those
  two things have nothing to do with each other anymore: the Three Laws and the iptables MASQUERADE
  rules do not belong in the same decisions/ folder. The hardware docs (overview.md, hardware.md,
  proxmox-setup, ubuntu-vm, vm-internet-bootstrap, archive/) are a coherent cluster that is *also* the
  least portable, most iMac-coupled content. Quarantine it. Prismo-the-product canon should not ship
  the 2017 iMac's systemd units.

- **`decisions/` is now 28 flat files spanning iptables-vs-bridging to Layer-3.5-pipeline-services.**
  Number-ordered flat files worked at 10 decisions. At 30+, with portability/commercial framing, this
  needs categorization. Either subfolder by domain (`decisions/infra/`, `decisions/architecture/`,
  `decisions/governance/`, `decisions/workflow/`) or — better — add a `category` frontmatter field
  (doctrine-service already scopes `duplicate_id` by category, so the concept exists) and let the
  index group them. A new contributor cannot navigate 28 flat decisions; the masterplan's "rapid
  onboarding" goal fails at the decisions folder.

- **Create `architecture/model.md` as the one canonical layer definition.** This is the gap Section 2
  named. One short doc: the four governance tiers, what each owns, the Activation/pipeline position,
  and the runtime-role taxonomy. Everything else (masterplan, facets, decisions) *links* to it instead
  of re-glossing it. This is the doc that should be loaded by the architect facet first, ahead of
  Decision 017.

- **Demote `v2-masterplan.md` and `v2-roadmap.md` to `architecture/history/` and write a short living
  `architecture/system.md` (system-level, not homelab) that reflects reality.** The masterplan was a
  founding vision doc; it has done its job and is now 60% annotations-correcting-the-body. Founding
  docs are valuable as history but dangerous as current reference. The phase model (0–7) is real and
  worth keeping in `v2-progress.md`; the 880-line prose vision is not load-bearing anymore. Stop
  annotating it; freeze it; point forward.

- **Retire the `phase1/` folder framing.** `architecture/phase1/` holds the canon-object-model,
  capability-contracts, lifecycle-semantics, retrieval-architecture. These are not "phase 1" artifacts
  — they are *current living contracts* (capability-contracts.md is literally enforced by the Service
  Rule at startup). Burying them under a phase number that implies "done and historical" hides
  load-bearing canon. Move them to `architecture/contracts/` or top-level `architecture/`. The phase
  number is metadata, not folder structure.

### Topic 2 — Facets

- **Highest-value missing facet: `ship` / `deploy`.** Every session that ends with a running service
  (prismo-ui shipped, context-server redeploys, pgvector migration) hits the same checklist: rebuild
  container, verify routes respond, check Service Rule passes, re-index, confirm topology. There is a
  `wrap-up` facet for *ending a session* but nothing for *putting a change into production*. Given
  Decision 032's "single docker compose up" and portability push, a deploy facet that loads the deploy
  runbook + Service Rule status + health endpoints is the obvious next one. Most "it broke after deploy"
  failures are exactly the framing failures facets exist to prevent (cf. the pgvector/doctrine bug that
  justified the `migrate` facet).

- **Second missing facet: `onboard` / `explain`.** The masterplan's stated north star is rapid
  onboarding and non-technical access (Shrey/Kyle). When someone asks "what is Prismo / how does X
  work / where do I start," the right behavior is to load system.md + the layer model + recent changes
  + active projects and explain at the *right altitude for the asker* — not dump architecture. This is
  also the facet that the future conversational web UI (Decision 031 second slice) will lean on. Build
  it now and the UI inherits a tested context bundle. Right now `architect` partially absorbs "explain
  the architecture" queries (per its activation examples) — that's a smell: architect is for *designing*,
  not *teaching*. Split the teaching load into `onboard`.

- **Weakness in existing facets: they all load homelab-only and assume a single project.** Every
  facet.yaml hardcodes `doc_type: homelab` / `project: homelab`. With prismo-ui, even, flight-planner,
  context-server all live and a multi-project future explicit in the masterplan, a `/review` or
  `/architect` session about prismo-ui loads the wrong canon. Facets need a project parameter (from
  session focus or `$ARGS`) rather than a hardcoded `homelab`. This is the facet-layer version of the
  Decision 032 portability problem — hardcoded scope.

- **The `migrate` facet is under-built relative to its origin story.** It exists because of a real
  production failure (pgvector → doctrine bug), but its SKILL.md is the thinnest of the five and its
  context loaders don't actually grep call sites or run the Service Rule — they load docs and tell the
  *model* to remember to do those things. That's back to behavioral enforcement, the exact V1 pattern
  the whole facet concept was meant to replace. A migrate facet that actually shells `grep -rn` for the
  changed symbol and hits `/doctrine/validate` before the model reasons would be structural. As-is it's
  a prompt reminder wearing a facet costume.

- **`sukuna` facet vs `scripts/sukuna.md` agent is a genuine fork that needs collapsing.** There are now
  two Sukunas: the facet (loads context for an interactive canon-maintenance session) and the script
  (this scheduled three-section pass). They overlap heavily and PI-013 — the doc that was supposed to
  reconcile them — got marked superseded without the reconciliation actually happening. Pick one: either
  the scheduled pass becomes a `SEQUENCE` activation (Decision 026 reserved that ActivationType for
  exactly this) that emits ReviewItems, or the script stays and the facet is just its interactive
  front-end. Right now they're drifting independently and both claim to be "Sukuna."

- **`SEQUENCE` is the most valuable unbuilt ActivationType and the scheduled-Sukuna pass is its
  killer first instance.** Decision 026 reserved FACET/SEQUENCE/MEMORY_PACK/CONSTRAINT/TOOLSET and only
  built FACET. A SEQUENCE is "a defined execution sequence" — which is exactly what sukuna.md is (read
  all canon → consistency → observations → thinking → emit). Building SEQUENCE to host the Sukuna pass
  would (a) prove the generic Activation abstraction was worth it, (b) close the drafts/ violation, and
  (c) give `wrap-up` and `deploy` a real home as sequences rather than facets-pretending-to-be-checklists
  (the wrap-up facet is a 5-step checklist — that's a SEQUENCE, not a cognition-shaping FACET). Several
  current "facets" are mis-typed: `wrap-up` and the proposed `deploy` are sequences, not facets.
