### Structural findings

- **`ANALYSIS_PROVIDER` env var has no decision record.** topology.py reads `ANALYSIS_PROVIDER` (line 141). The env var controls which provider handles analysis_runtime. Decision 023 documents `ANALYSIS_MODEL`. Decision 024 documents `SYNTHESIS_PROVIDER`. Neither Decision 023 nor 024 documents `ANALYSIS_PROVIDER` as a valid control surface. It exists in code but has no formal decision record, no documented valid values, and no entry in capability-contracts.md's routing table.

- **`coding_runtime` is hardcoded in topology.py, not config-driven.** Lines 121–129 of topology.py hard-wire `execution_mode="subscription_cli"`, `provider="anthropic"`, `model="current_session"` — no env var lookup. Decision 025 explicitly states coding_runtime is a role not an identity. The implementation violates this principle at the code level. A `CODING_RUNTIME_MODE` / `CODING_RUNTIME_PROVIDER` env var path (even defaulting to the current values) would make the invariant structural rather than aspirational.

- **"Capability Contracts" names two different concepts.** `docs/architecture/phase1/capability-contracts.md` defines "Capability Contracts" as stable HTTP interface definitions between layers (`retrieve()`, `synthesize()`, etc.). Decision 025 defines "Capability Contracts" as what a runtime role advertises it can do (`supports_tool_use`, `low_latency`). topology.py uses `CAPABILITY_CONTRACTS` as the variable name for the second concept. These are not the same thing. As both grow, the shared name will produce genuine confusion for new contributors and in MCP query responses.

- **v2-progress.md `capability-contracts.md` Phase 1 deliverable description is outdated.** Phase 1 lists the capability-contracts.md deliverable as covering `retrieve / query / summarize / synthesize / plan / codegen / review / analyze / hydrateSession / generateOperationalBrief`. This list predates Phase 3 additions (captureSignal, ReviewItem flow, runtime topology) that are now documented in the same file. The Phase 1 summary paragraph under Phase 1 should note that the doc has grown through Phase 3.

- **context-server decisions 001–014 have no YAML frontmatter.** Homelab decisions 020–025 carry full frontmatter (`id`, `title`, `status`, `record_type`, `date`). context-server decisions carry none. When Pydantic frontmatter validation ships and doctrine-service runs, all 14 context-server decisions will surface as metadata-reconciliation findings. Worth flagging before that work starts so the false-positive rate is understood.

## 2. Observations

- **The runtime topology is an env-var mirror, not operational state.** Decision 025's strongest architectural claim: "The runtime topology is not a config file. It lives in the same SQLite (soon PostgreSQL) operational state." topology.py builds the topology from `os.getenv()` on every request. There is no database write, no persistent topology record, no `POST /runtime/topology` write path. The "which runtime touched this ReviewItem?" audit query Decision 025 promises is impossible with the current implementation. The operational-state topology is the key un-built piece of the 025 spec.

- **CapabilityRegistry and topology build independently — they can drift.** The CapabilityRegistry (registry.py) holds what providers are actually *registered*. topology.py describes what is *configured via env vars*. These two systems are not connected. A caller who reads the topology gets the intent; a caller who uses the registry gets the runtime reality. If an env var says `SYNTHESIS_PROVIDER=anthropic` but the startup sequence registered a ClaudeCodeSynthesisProvider, the topology will lie. No consistency check exists between them.

- **"General Collaboration Runtime" from the masterplan has no topology slot.** Decision 025 names four roles: coding_runtime, synthesis_runtime, analysis_runtime, embedding_runtime. The masterplan Layer 3 explicitly calls out a "General Collaboration Runtime" for teammates (Shrey/Kyle) — low cost, high availability, context-integrated. When a web UI or WhatsApp adapter ships, Shrey or Kyle interacting with Prismo will need a named runtime role. That role has no entry in topology.py, no authority definition, no capability contract. It should be declared as a named future slot before any Layer 4 work begins.

- **The observation-week tuning loop from Decision 021 has no canon record of findings.** Decision 021 mandated an observation week with empirical tuning of synthesis quality gates. `prismo metrics` ships the numbers. But there's no decision, annotation, or proposed-idea capturing what the observation week actually found and what gate adjustments (if any) resulted. Before Phase 4 work deepens, a closure record is needed — either "observation week confirmed default thresholds are correct" or "thresholds adjusted to X based on Y pattern." The loop currently doesn't close to canon.

- **LiteLLM is declared as the api-mode implementation detail but hasn't been wired.** Decision 025 positions LiteLLM "inside the provider implementation, not at the interface layer." The 2026-05-26 research draft triage confirms the plan: wire LiteLLM into AnthropicSynthesisProvider/AnthropicAnalysisProvider. Until this wiring lands, api-mode providers are raw Anthropic SDK calls, model-swappability via string change doesn't work, and the three-concept separation's promise (change provider without touching callers) isn't fully realized for the api execution mode.

- **The "two-provider completion criterion" from Decision 024 has no audit surface.** Decision 024 requires every capability to have ≥2 provider implementations before it's considered complete. `embed` (EmbeddingProvider) has only one: SentenceTransformers. There is no registry query, no capability-contracts.md field, and no doctrine-service rule that enforces or surfaces this criterion. It's behavioral compliance right now, not infrastructure.

- **Decision 022 (adapter ontology drift) is the right diagnostic but has no link to the real drift that already exists.** The analysis_runtime default model inconsistency (Haiku in topology.py, Sonnet in Decision 023) is exactly the class of problem Decision 022 was written to prevent. The irony: the Layer 3 implementation that shipped concurrently with the decision that identifies this smell contains an instance of the smell. Decision 022 should either annotate this or the fix to topology.py should reference Decision 022.

## 3. Thinking

- **The "topology in operational state" gap is blocking before it's obvious it's blocking.** Right now, topology is computed from env vars at query time. This works fine. The problem arrives when: (1) a ReviewItem is generated — you want to know which runtime_role, which execution_mode, which model produced it; (2) topology changes — you want to know the before/after. Neither is possible without a write path. The PostgreSQL migration is the right time to add a `runtime_topology_snapshots` table: write a snapshot at startup (and on config change), query it for audit. Don't build a full dynamic routing system — just make the topology durable. One table, one write on startup.

- **"Capability Contracts" must be split into two terms before it becomes canonical confusion.** Proposal: rename the Phase 1 doc concept to "Service Contracts" (the stable HTTP interfaces between layers), and use "Runtime Capabilities" for what Decision 025 and topology.py call capability contracts (the tags a runtime role advertises). Both concepts are correct and valuable — they just can't share a name. The Phase 1 doc rename is a one-line diff in capability-contracts.md's header and a handful of references in decisions. Do this before pgvector lands, since pgvector work will touch capability-contracts.md.

- **The AnalysisProvider Protocol stub is too thin to be a real contract.** `analyze(analysis_type, description, context="") -> dict` returns `dict`. The actual analysis output is a structured `AnalysisResult` with `findings`, `recommendation`, `sources`, `confidence`, `review_item_id`. The Protocol declares a weaker contract than the implementation. When a second AnalysisProvider implementation (Ollama-based?) ships, the implementor will see `-> dict` and not know what fields are required. Either define an `AnalysisResult` TypedDict/dataclass and make the Protocol return it, or accept that the Protocol is documentation-only and remove the `@runtime_checkable` decorator which implies structural enforcement.

- **pgvector kills three problems at once, not one.** The synthesis-service justification for pgvector is "replace ChromaDB." The full argument is stronger: (1) removes ChromaDB container + volume + known collection.get() bug; (2) enables topology snapshot table in the same Postgres instance (closes the "topology in operational state" gap from Decision 025); (3) workflow-state-service SQLite → Postgres migration becomes natural (the same instance, not a separate migration project). Present pgvector as the unified persistence migration, not just a vector store swap.

- **The `collaboration_runtime` slot should be declared now.** Shrey and Kyle are named in the masterplan. When they get access (web UI, WhatsApp), their intelligence slot should be named. Declaring it now as `status: reserved` in topology.py (`ROLE_DEFINITIONS`) does two things: (1) it proves the architecture before anyone builds on it; (2) it forces the authority boundary conversation now rather than during implementation. What is a collaboration_runtime permitted to do? Probably: read canon, ask questions, capture signals — but NOT write ReviewItems directly. That authority boundary matters and should be decided before the surface is built.

- **The three-concept separation (Role / Mode / Provider) is coherent with the masterplan but more useful than the masterplan's framing.** The masterplan describes Layer 3 in terms of "Runtime Categories" (Coding, General Collaboration, Specialized). Decision 025's three axes are orthogonal to that category taxonomy — they're a governance vocabulary, not a classification. The masterplan categories describe *what* runs; the three concepts describe *how to manage* what runs. Both are needed. The masterplan Layer 3 description should be annotated to cross-reference Decision 025 as the governance refinement.

- **Pydantic frontmatter validation is more load-bearing than it looks.** The pitch is "validate frontmatter before indexing." The real value: once every canon doc has machine-readable `status`, `id`, `superseded_by`, the CapabilityRegistry's "two-provider completion criterion" becomes a doctrine-service rule (find any capability in interfaces.py with < 2 registered providers, surface as ReviewItem). Pydantic validation is the prerequisite for making Decision 024's completion criterion enforceable by infrastructure rather than discipline. Frame it this way in the proposed-idea.

- **The env var naming migration from Decision 025 hasn't started.** Decision 025 declares: `SYNTHESIS_PROVIDER=claude-code` → `SYNTHESIS_ROLE_MODE=subscription_cli` + `SYNTHESIS_ROLE_PROVIDER=anthropic`. topology.py still reads the old `SYNTHESIS_PROVIDER` key. The migration is described as non-breaking ("old env vars are supported until deprecated"). But the new vars aren't read anywhere yet, so the three-axis model exists only in the decision doc. Before adding more runtime roles or capabilities, the new var convention should be implemented (even if the old vars still work as fallback). Otherwise the three-concept separation is a documented abstraction with a one-concept implementation.

- **What does "prismo topology" as a CLI command look like?** `prismo session current` shows workflow state. `prismo review` shows ReviewItems. There's no `prismo topology` that would show the runtime intelligence topology. Given that the whole point of `GET /runtime/topology` is observability, a CLI wrapper seems natural and low-effort. The prismo script already wraps every other Layer 3 surface. This is a 10-line addition.

- **The synthesis observation week + the runtime topology together enable something neither enables alone: cost traceability.** Once topology snapshots are in Postgres and each ReviewItem carries a `produced_by_role` field (synthesis_runtime, analysis_runtime), it becomes possible to answer: "how much did observation week cost in API calls, and which model tier did most of the work?" That number is currently invisible. It matters for deciding when to flip synthesis to local mode. This is the practical argument for doing topology-in-operational-state before pgvector, not after.
