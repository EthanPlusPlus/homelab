## 2. Observations

- **013's frontmatter overstates the doctrine-service dependency, understates the synthesis gate dependency.** Any retrieval query on proposed-idea 013's lifecycle status will read "defer until doctrine-service Day-1 lands" and infer that Sukuna refactor is blocked on Phase 3 infrastructure. The actual binding constraint is observation-week synthesis calibration, which closes first. This matters: if doctrine-service ships before observation week completes, someone could reasonably interpret 013 as unblocked when synthesis gates are still untested.

- **014 uses the word "synthesis extension" but describes a fundamentally different input bus.** Decision 015 defines `SynthesisProvider.synthesize(task, context, max_tokens)` — internal context in, prose out. External synthesis requires web, GitHub, and paper retrieval — outside the Layer 2 boundary, consuming different APIs, with different latency and failure characteristics. Naming it a "synthesis extension" will cause it to be scoped into synthesis-service, when architecturally it needs its own input provider abstraction (or its own service). The naming is doing architectural work it shouldn't.

- **The 3-mode synthesis table in 014 has no canonical home.** It belongs as an annotation to Decision 021 (which defines the synthesis architecture) or as a separate decision once mode 1 is stable. As a section inside a proposed-idea, it has no architectural authority — proposed-ideas are not consulted as binding specs. If the 3-mode model shapes how synthesis-service is built, it needs to be in a decision.

- **`~/CLAUDE.md` Session Bootstrap warm-up (step 2) is not annotated as superseded by Decision 016.** Decision 016's title is "session-hydration-replaces-warmup." workflow.md step 0 reflects the V2 form. CLAUDE.md is still the primary behavioral instruction file for Claude sessions — the warm-up instruction is actively executed when `current_focus` is unset and the UserPromptSubmit hook fires. There's no pointer from CLAUDE.md to Decision 016 or workflow.md explaining that V2 structural hydration is the intended mechanism. Someone reading CLAUDE.md has no way to know this section is V1 debt.

- **The canon discipline section in `~/CLAUDE.md` (Phase 1/Phase 2 MCP ritual) has no annotation identifying it as a behavioral ritual scheduled for structural replacement.** The V2 masterplan explicitly names "workflow enforcement is advisory" as a V1 problem. The Phase 1/Phase 2 ritual is the clearest surviving instance. Unlike the warm-up (which has a structural replacement in Decision 016), the canon discipline ritual does NOT yet have a structural replacement — synthesis preflight + doctrine validation are the eventual replacements, but neither is scoped to cover the "query MCP before proposing" gate specifically.

- **v2-masterplan.md Layer 4 section names Web UI as the primary interface.** Decision 021 explicitly inverts this: CLI ships first, web does NOT ship first. The masterplan is a foundational doc that people read for architectural orientation; someone reading it cold gets the wrong priority sequence. The masterplan predates Decision 021 and has no annotation marking the Layer 4 priority inversion.
