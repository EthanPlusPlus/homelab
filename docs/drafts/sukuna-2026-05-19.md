## 2. Observations

- **013's frontmatter overstates the doctrine-service dependency, understates the synthesis gate dependency.** Any retrieval query on proposed-idea 013's lifecycle status will read "defer until doctrine-service Day-1 lands" and infer that Sukuna refactor is blocked on Phase 3 infrastructure. The actual binding constraint is observation-week synthesis calibration, which closes first. This matters: if doctrine-service ships before observation week completes, someone could reasonably interpret 013 as unblocked when synthesis gates are still untested.

- **014 uses the word "synthesis extension" but describes a fundamentally different input bus.** Decision 015 defines `SynthesisProvider.synthesize(task, context, max_tokens)` — internal context in, prose out. External synthesis requires web, GitHub, and paper retrieval — outside the Layer 2 boundary, consuming different APIs, with different latency and failure characteristics. Naming it a "synthesis extension" will cause it to be scoped into synthesis-service, when architecturally it needs its own input provider abstraction (or its own service). The naming is doing architectural work it shouldn't.

- **The 3-mode synthesis table in 014 has no canonical home.** It belongs as an annotation to Decision 021 (which defines the synthesis architecture) or as a separate decision once mode 1 is stable. As a section inside a proposed-idea, it has no architectural authority — proposed-ideas are not consulted as binding specs. If the 3-mode model shapes how synthesis-service is built, it needs to be in a decision.

- **`~/CLAUDE.md` Session Bootstrap warm-up (step 2) is not annotated as superseded by Decision 016.** Decision 016's title is "session-hydration-replaces-warmup." workflow.md step 0 reflects the V2 form. Someone reading CLAUDE.md has no way to know this section is V1 debt.

- **The canon discipline section in `~/CLAUDE.md` (Phase 1/Phase 2 MCP ritual) has no annotation identifying it as a behavioral ritual scheduled for structural replacement.** The V2 masterplan explicitly names "workflow enforcement is advisory" as a V1 problem. ## 3. Thinking

- **014 is correctly sequenced but incorrectly named.** "External synthesis" implies a synthesis capability. What 014 describes is an external *signal provider* — a tool that fetches structured research from the open web and emits a ReviewItem. Synthesis is what happens to signals. Calling it "external synthesis" will cause it to be built into synthesis-service, when it should be built as an input adapter that feeds synthesis-service the same way captures do. Rename to "external signal provider" or "prior-art research provider" before it touches a build queue.

- **The observation-week freeze is the right call for 014, but the framing exposes a gap: there's no mechanism to track "manual instances of this pattern" during the week.** 014 says "use observation week to confirm the trigger pattern actually shows up in captures." But who's watching for it? If prior-art moments happen in sessions but the engineer forgets to capture them, the observation evidence doesn't accumulate. Add a capture prompt or a session note: "when you google something before building, capture it."

- **The 3-mode model should be an annotation on Decision 021, not a section in 014.** Decision 021 is the synthesis architecture doc. The 3-mode table belongs there as a "future architecture directions" annotation — with a clear note that modes 2 and 3 are unbuilt and depend on mode 1 being stable. Leaving it in 014 gives it zero architectural visibility and it will be ignored when synthesis-service is extended.

- **Alternatively: promote 014 to a decision (status: proposed, not active) immediately.** A proposed decision has more architectural authority than a proposed-idea. The 3-mode model, the trigger pattern, and the `prior-art-report` ReviewItem type extension together form a coherent architectural direction that warrants decision-level treatment. A proposed-idea won't be consulted when someone extends synthesis-service six months from now.

- **The CLAUDE.md canon discipline ritual has no structural replacement at current build state — but it has a clear retirement trigger.** The retirement condition should be explicit in CLAUDE.md: "retire this section when doctrine-service preflight check covers structural canon conflicts + synthesis-service quality gate covers significance thresholds." Write that trigger condition into the file now, so the cleanup has a target rather than accumulating as indefinite technical debt.

- **The Session Bootstrap warm-up in CLAUDE.md (step 2) should be removed now, not deferred.** Decision 016 ships structural hydration. workflow.md step 0 is the V2 form. The warm-up instruction in CLAUDE.md is actively contradicting structural hydration for sessions that CLAUDE.md governs. This isn't deferred cleanup — it's an active inconsistency producing different behavior depending on which file a session reads first. Clean it: remove step 2 from CLAUDE.md, replace with a pointer to workflow.md step 0.

- **The Phase 1/Phase 2 MCP ritual in CLAUDE.md is behavioral, but it's also doing something the structural layer genuinely can't do yet: it gates on *intent*.** The hook system can detect events (message submitted, session started) but can't detect "is this message proposing something?" That semantic classification is exactly what synthesis-service will eventually do automatically. Until then, the ritual is the only mechanism. But calling it "V1 technical debt" without acknowledging this leaves the gap unnamed. The honest framing: it's an *intentional behavioral bridge* until synthesis-service's quality gate subsumes it.

- **Decision 020's framing in the canon creates a false dependency for 013.** 013's notes say "defer until doctrine-service Day-1 lands." But doctrine-service Day-1 (per Decision 020) is scoped for staleness eval + metadata reconciliation + supersession-minimal. None of those capabilities are actually required for the Sukuna refactor. The Sukuna refactor needs synthesis-service to accept structured inputs and emit ReviewItems — which already shipped. The doctrine-service dependency in 013 is vestigial; the actual blocker is quality gate calibration.

- **There is no proposed-idea or decision tracking the ReviewItem schema extension process.** 014 adds `prior-art-report`. Someone else will add another type. The Decision 021 schema is de facto extensible but has no governance: no process for proposing new artifact types, no versioning discipline, no doc that lists "these types exist, these are planned." Before 014 builds, there should be a schema extension decision or at least a tracked proposed-idea that 014 can reference instead of casually adding a new type in its JSON block.
