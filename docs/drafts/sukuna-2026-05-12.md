Priority 1 (architectural refactor), Priority 2 (context-server v2), Priority 3 (runtime abstraction) — all have no tracking artifact. The masterplan is currently an orphaned document, not a roadmap.

- **context-server/decisions/005 and Decision 007 are in direct conflict** in their description of the indexer root path. 005 says `PROJECTS_PATH`; 007 says `CANON_PATH`. Any agent reading both gets a contradictory picture of how doc discovery works. A decision is supposed to be immutable once adopted — 005 needs an addendum, not a rewrite.

- **"Start sessions with hi" is a ghost instruction.** It appears in memory/workflow.md and several project CLAUDE.md files but has no decision record and contradicts the actual Decision 012 text ("first user message of every session," no greeting requirement). It's being replicated into new project scaffolds (flight-planner, even) as if it's canon.

These should be closed, promoted, or formally reopened — but the folder reflects none of these transitions. The proposed-ideas/ folder is quietly becoming a graveyard of mixed-lifecycle items.

The add-vm-user.md runbook has a known blocker (prismo new-machine doesn't work on the VM). Either the runbook needs fixing or the onboarding is stalled. The masterplan will accumulate drift the same way decisions/005 did if its dependencies aren't tracked.

## 3. Thinking

Direction: Prismo V2 redesign lens — evaluate V1 canon through v2-masterplan.md

**(1) What in V1 is genuinely load-bearing and must survive intact:**

- **canon/ structure (markdown + git + filesystem)** — This is Layer 1. The entire V2 superstructure is built on top of this. Non-negotiable.
- **The ~/canon/ / ~/projects/ separation (Decision 007)** — The code/knowledge split is a genuine architectural insight, not an implementation accident. V2 builds on this topology.
- **context-server as a service (not embedded in Claude)** — The separation of retrieval from runtime is the embryo of the Layer 3 abstraction. Keep the service; expand it toward Layer 2 capability contracts.
- **The drafts-only write constraint (Decision 003)** — "Truth is curated, not automated." This is the structural implementation of V2's "Humans retain authority" principle. It must survive — V2 expands automation without removing the review gate.
- **The triggered canon discipline (Decision 012 over 009)** — "Every thought" failed; triggers work. V2's workflow-state-service will eventually enforce this structurally, but the trigger logic itself is correct.
- **Decisions/ as immutable adopted records** — The lineage of supersession chains (009 → 011 → 012) is precisely what V2's doctrine-service would manage. The pattern is right; the tooling is manual.

**(2) What is accidental complexity / evolutionary cruft to drop in V2:**

- **The warm-up conversation ritual** — The "ask one question, query MCP, repeat" loop is compensating for the absence of a workflow-state-service that would produce an operational brief automatically. In V2, session start produces a context bundle from Layer 2, not a conversation dance. This pattern should not be replicated into the V2 web UI.
- **The git merge ceremony (Decision 008)** — Merging the docs branch into master after every doc commit is friction with no V2 justification. In V2's world, docs and code histories don't need to be unified in a single branch; they're indexed independently. Drop it as a requirement.
- **Stop hook + UserPromptSubmit hook** — Both are behavioral patches for the absence of structural enforcement. V2's workflow-state-service makes them unnecessary. Don't port them forward.
- **The prismo bash script** — Band-aid over missing Layer 2 orchestration. In V2, `prismo new-project` is a Layer 2 API call that provisions the substrate, not a bash script that drifts. The script's existence is a sign of missing infrastructure, not a feature to preserve.
- **The "pull-before-commit" parallel session protocol** — Manual concurrency coordination. V2's workflow-state-service tracks active work; the session-start check is a Layer 2 capability.

**(3) What is tightly Claude-coupled and needs abstraction:**

- **CLAUDE.md as the governance layer** — The entire workflow, discipline, and operational behavior is expressed as Claude Code instructions. Other runtimes (Shrey's future web UI, local models) have no way to load these. V2 must move governance into Layer 2 service contracts, with CLAUDE.md reduced to a thin adapter for the Claude runtime.
- **MCP tool surface (search_docs, get_doc_section, etc.)** — These are designed for Claude Code ergonomics, not capability contracts. V2's Layer 3 contract is `retrieve(query, project)` — the MCP tool names are implementation details of one runtime. Abstract the capability; keep MCP as one transport.
- **Sukuna as a Claude agent definition** — The sukuna.md format is Claude-specific. V2's Sukuna is a scheduled Layer 2 service (or a callable maintenance endpoint), not a Claude agent file. The current form is a prototype; don't ossify it.
- **Session bootstrap warm-up logic** — "Ask one question, query MCP, repeat 2–4 times" is Claude conversation design. The web UI for Shrey/Kyle doesn't have this loop — it has a dashboard. The information produced by the warm-up (what's the user building, what's relevant canon) should be a Layer 2 context bundle, not a conversation routine.
- **The hooks system (Stop, UserPromptSubmit)** — Claude Code-specific. Not portable to any other runtime. Treat as temporary scaffolding only.

**(4) Proposed-ideas alignment with V2 priorities vs. close/supersede:**

ALIGN — these are V2 work:
- **006 — Mobile gateway** → Layer 4. WhatsApp is explicitly named as V2's nearest-term interface. This is the most aligned proposed-idea in the entire canon. Should be elevated to a formal V2 initiative.
- **003 — Embedding model upgrade** → Priority 2 (context-server v2). The upgrade is a prerequisite for "metadata-aware retrieval" and "stale-content surfacing." Don't defer indefinitely — attach it to the context-server v2 work.
- **002 — Gitea migration** → Priority 3 / Layer 1 durability. Reducing GitHub dependency is a V2 concern. Conditions (stability, redundancy) are still not met, but the direction is right. Keep open.
- **context-server/003 — Log/service state retrieval** → V2 workflow-state-service. The "runtime context" concept in this proposed-idea is a partial specification of what Layer 2's workflow-state-service does. Should be reframed as a V2 work item.
- **008 section 2 — Interaction observer** → V2 synthesis-service. Watching sessions for un-canonized decisions is exactly what V2 describes as "contribution capture." This is open with no home — needs a proposed-idea of its own.
