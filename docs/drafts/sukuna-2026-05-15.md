## 2. Observations

- **Workflow-state-service exists but is invisible from the consumer side.** Decision 009 shipped 6 new MCP tools (start_session, update_focus, create_workstream, etc.) but memory/mcp.md doesn't list them and memory/workflow.md doesn't reference them. The structural replacement for the bootstrap ritual is built but the human-facing instructions still describe the manual warm-up. Nobody is calling these tools yet.
- **v2-masterplan.md and v2-roadmap.md have no decision records.** They are the most architecturally consequential documents in canon — defining 4 layers, 7 programs, 5 V2 priorities — but they live as `architecture/` files with no `decisions/` entry adopting them. They cannot be superseded cleanly because they were never formally adopted.
- **Phase 0/1 deliverables are complete but undocumented against the roadmap.** The roadmap lists Phase 0 deliverables as `v1-survivors.md, v1-coupling-map.md, v1-deprecated-patterns.md, v1-service-map.md` — all present in v2-audit/. Phase 1 deliverables are `canon-object-model, capability-contracts, lifecycle-semantics, retrieval-architecture` — all present in phase1/. Phase 2 shipped context-server v2 + workflow-state-service + brief engine. But there is no doc tracking this progress against the roadmap. Progress is only visible in recent-changes.md, which rolls off after 10 entries.
- **"Hydrated session start" is built but not used.** ContextBundle is live with core_operational_state, active_doctrine, recent_context, etc. — yet the CLAUDE.md instruction still tells Claude to manually read recent-changes.md and run a 2–4 turn warm-up. The roadmap explicitly says "the runtime starts hydrated" — Layer 2 already produces this, but Layer 1 instructions haven't been updated to consume it.
- **Lifecycle semantics not enforced anywhere.** lifecycle-semantics.md defines staleness rules (proposed >90d, experimental >60d). `get_stale_items` exists. But no scheduled job runs it, no Sukuna pass uses it, and no notification surfaces stale items to humans. The detection capability is built; the enforcement loop is not closed.
- **"Sukuna" appears in two places with different scopes.** proposed-idea 009 describes the current Claude-agent prototype. v2-masterplan / v2-roadmap Initiative 2.5 describes "Sukuna v2" as a Layer 2 maintenance intelligence system. These are different things sharing a name — the prototype writes drafts; the v2 service runs scheduled passes and reconciles metadata. No file states this clearly.
- **SynthesisProvider abstraction is mentioned in recent-changes but has no canon record.** "SynthesisProvider abstraction (mirrors EmbeddingProvider)" appears in recent-changes.md as part of Phase 2 brief engine work — but there is no decision, no architecture doc, no proposed-idea describing it. This is exactly the kind of architectural choice that should have a decision record.

## 3. Thinking

**The biggest gap right now is not technical — it's that V2 progress is undocumented except in recent-changes.md.** Phase 0, 1, 2 are arguably done. Nothing in canon says so. recent-changes.md will roll off these entries within 10 commits. Write `architecture/v2-progress.md` as a living index against the roadmap. Without it, the next contributor can't tell what's built from what's planned.

**Workflow-state-service is the most important thing shipped this month and nobody knows how to use it.** The bootstrap ritual in memory/workflow.md is now wrong — it describes a manual warm-up that the service was built to replace. Update memory/workflow.md to: (1) call `start_session` on session start, (2) read the ContextBundle from `get_context`, (3) only warm up if `core_operational_state.current_focus` is empty. Otherwise, you built a Layer 2 service and kept doing Layer 1 by hand.

**Phase 3 should not start until lifecycle enforcement is actually live.** doctrine-service is the natural next step but the prerequisite (lifecycle-aware retrieval) shipped without a closed loop — `get_stale_items` is callable but never called. Build the trigger before building more services. Cron + Slack/email digest beats another microservice.

**The doctrine-service / synthesis-service / Sukuna v2 trio described in the roadmap is over-scoped.** Three Phase 3 services with overlapping responsibilities (stale detection, supersession tracking, drift detection, contradiction surfacing all show up in two of three). Collapse them. Either: (a) one canon-maintenance service that owns lifecycle, supersession, drift, and synthesis; (b) doctrine-service is data (graph, lifecycle queries, no LLM) and synthesis-service is the LLM consumer. Don't ship three services with fuzzy boundaries — that's V1's accidental complexity in V2 clothing.

**SynthesisProvider as a separate abstraction from EmbeddingProvider may be wrong.** Both are model-runtime adapters. The v2-masterplan describes Layer 3 as a single Runtime Provider Interface with capability contracts (`retrieve`, `summarize`, `synthesize`, `embed`). Two parallel provider abstractions feel like premature decomposition. Reunify under one RuntimeProvider that exposes capabilities, then have context-server depend on `runtime.synthesize()` and `runtime.embed()` against the same registry.

**Sukuna v2 should not exist as a named agent.** The current sukuna.md is a Claude agent that writes drafts. v2-roadmap describes Sukuna v2 as a Layer 2 service. The right move is not "rebuild Sukuna as a service" — it's "delete Sukuna as a concept." Split it into: (a) `get_stale_items` + scheduled trigger (already mostly built) for mechanical maintenance, (b) a generic `synthesize(canon)` capability called by a cron or a human button. The agent-with-a-name framing leaks anthropomorphic clutter into a system that's supposed to be capability-contract-based.

**The web UI is the actual blocker, not Phase 3.** v2-roadmap Phase 5 is "human interaction layer." Shrey and Kyle don't have any way to use this. The longer you build Layer 2 cognition without a Layer 4 surface, the more you'll over-engineer Layer 2 around imagined Claude-Code-shaped use cases. Build the smallest possible web UI (auth, one project view, conversational query against ContextBundle, contribution capture) next. Phase 3 can wait — Phase 5 unblocks the team.

**The Claude-coupling problem is getting worse, not better, despite v2 work.** Every new Phase 2 capability ships as an MCP tool. MCP is Claude Code's protocol. The capability contracts in phase1/capability-contracts.md describe `retrieve()`, `summarize()`, `synthesize()` — none of which is the surface the new tools expose. workflow-state-service shipped with 6 MCP tools and 0 HTTP endpoints documented in capability-contracts. You're paving the runway with the wrong material.

**Decision-record discipline has lapsed.** v2-masterplan, v2-roadmap, EmbeddingProvider, SynthesisProvider, ContextBundle priority stratification, workflow-state-service's three-object model — none has a homelab/decisions/ entry. These are exactly the choices a contributor needs supersession chains for. The "immutable adopted record" pattern from V1 is being skipped just as V2 makes more consequential decisions. Backfill: write 013 (V2 masterplan adopted), 014 (workflow-state-service architecture), 015 (synthesis provider abstraction).

**The roadmap underestimates the cognitive load of having two parallel canon directories.** ~/canon/homelab/ has v2-masterplan, v2-roadmap, phase1/, v2-audit/. ~/canon/context-server/ has decisions 008 and 009 implementing v2. The v2 work spans two repos with no cross-reference index. Phase 3 will span three or four. Consider a top-level `~/canon/prismo-v2/` for the masterplan, roadmap, and program tracking — distinct from any single component's canon. Right now the masterplan lives in homelab/ as if Prismo V2 is a homelab subproject. It's the other way around.
