## 3. Thinking

- **Decision 026 is architecturally correct but the pipeline hasn't shipped yet, and the SKILL.md it depends on is already calling the pipeline "pre-assembled."** The SKILL.md reads: "The Architect Facet context has been pre-assembled by the pipeline service." The pipeline doesn't exist yet in code. `pipeline/` module is in "Remaining Phase 4 work." The SKILL.md was written for the future state and deployed into the current state. Any time `/architect` is invoked today, the model is told that context was pre-assembled when it wasn't. This is not a minor inconsistency — it's the model being told to rely on infrastructure that doesn't exist, and the fallback is the manual MCP steps the doc says to use only when "context-server may be unreachable." Reorder: build `pipeline/` first, then deploy the SKILL.md that depends on it.

- **The synthesis loop has been running for months with no auto-Sukuna pass.** Sukuna runs are still manual invocations. Proposed-idea 009 says "Cadence: on-command for now; scheduling TBD." There's a daily cron for `prismo stale`. There is no scheduled Sukuna run. Given that Sukuna is the only consistency-pass mechanism and doctrine-service hasn't shipped, the current situation is: structural consistency is maintained only when someone manually invokes Sukuna. That's a behavioral dependency, not structural enforcement — the exact thing V2 set out to eliminate.

- **The "observation week" framing has aged into an abuse of the word "week."** Decision 021 mandated "observe for one week." The loop has been running since mid-May. That's two weeks at minimum. The word "week" in the decision is now a governance liability — it implies the observation period is ongoing indefinitely, which means the post-observation work (doctrine-service, Sukuna refactor, etc.) is perpetually sequenced behind a gate that nobody has formally closed. Either close the observation week with a canon record (write the damn findings) or rename it "observation period" and define a closing condition.

- **The pipeline hook injecting `[FACET: architect]` on every relevant prompt is ambitious but creates a new billing surface nobody has modeled.** The current `prismo session ensure` hook fires on every prompt and is free (it's a CLI call, no AI involved). Decision 026's pipeline hook calls `POST /pipeline/process` which runs Stage 1 (keyword filter, free) and potentially Stage 2 (embedding comparison, near-free). But if proposed-idea 016 (routing_runtime — small model as activation router) ships and replaces Stage 2 with a Haiku call, the per-prompt cost jumps to ~$0.001 per classified prompt. Across a working day of 200 prompts, that's $0.20/day, $5/month. Not catastrophic. But it hasn't been estimated, stated, or accepted. Cost visibility is a Decision 023 discipline (two AI tiers, explicit). The pipeline tier should also be explicit.

- **The Response Processor slot in Decision 026 is the most impactful unbuilt component no one is talking about.** The Activation Router and Context Assembler are useful but they're context injection — a refinement of what already exists. The Response Processor (outbound processing — capture detection from model output) would close the loop that currently requires human discipline: capturing insights before they disappear. Right now, Claude is responsible for calling `capture_signal` inline. The Response Processor would detect architectural decisions in model output automatically and surface them as `CaptureSuggestion`. This is the piece that converts "model generates insight" into "system automatically nominates it for canon." It's listed as "reserved for Phase 4 completion" but it's arguably more architecturally important than the router.

- **Proposed-idea 013 (Sukuna as synthesis consumer) still defers for doctrine-service Day-1 and observation-week calibration. Both of those deferred blockers may now be met or near-met.** Observation week has run. Doctrine-service has a complete scope in Decision 020 and is next in the Phase 3 queue. If both blockers clear in the next session, the sequencing reason for Sukuna writing drafts disappears. The current run is writing to `drafts/` — a practice Decision 021 explicitly retired. The architecture says Sukuna should emit ReviewItems. The blockers that prevented this were pragmatic, not principled. They may no longer apply. Worth checking.

- **The `context-server/docs/` canon has structurally diverged from the `homelab/docs/` governance conventions.** homelab decisions carry full frontmatter, standard sections, record_type. context-server decisions carry none of this. As doctrine-service matures and runs against both, it will produce a constant stream of metadata-reconciliation findings from context-server that are noise, not signal. There should be a one-time "bring context-server docs to standard" session before doctrine-service ships — otherwise its first output is a 14-item false-positive report.

- **Nobody has written down what Prismo will look like when a second human contributor (Shrey or Kyle) actually uses it.** The masterplan mentions them. Decision 021 mentions "per-contributor profile-based synthesis tuning." But there's no proposed-idea, runbook, or architecture note for "Shrey's first session with Prismo." What does he actually touch? `prismo review`? A web UI that doesn't exist? WhatsApp that doesn't exist? The onboarding story for non-Ethan contributors is entirely unwritten. Given that multi-user support is a V2 founding principle, the absence of any concrete contributor-onboarding artifact is the biggest silent gap in the whole roadmap.

- **The "Prismo is not a homelab" ontology tension has been "resolved" but the resolution is awkward.** The homelab/prismo ontology doc says the rename to `~/canon/prismo/` is deferred "until web UI / new-contributor onboarding provides a forcing function." That forcing function — the web UI / Shrey onboarding — is getting closer. When it arrives, explaining to a new contributor that "Prismo's canon lives at `~/canon/homelab/`" is going to require apology. The rename cost (updating symlinks, re-indexing, updating cross-references) is actually low. The reason it keeps getting deferred is unclear — this is habit, not architecture.
