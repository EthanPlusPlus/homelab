## 3. Thinking

- **Sukuna writing to `drafts/` is the only Decision 021 violation that writes a new canonical artifact on every single run.** The proposed-idea 013 blocker chain (wait for doctrine-service, wait for observation-week closure) is real but not the actual obstacle. The simpler path: emit each section-bullet as a ReviewItem directly via `POST /review/queue`, skip the markdown draft. The synthesis quality gates are calibrated enough at 13 days. The blocker isn't infrastructure — it's that no one has built the Sukuna-to-ReviewItem path. That's a few hours of work, not a dependency on doctrine-service.

- **The observation week produced real signal (wrong-artifact-type dominated rejections) that was captured in a recent-changes.md entry but never formalized.** That pattern — synthesis emitting wrong artifact type as the dominant failure — should have become a threshold adjustment, an annotation on Decision 021, or a proposed-idea about artifact-type pre-classification. Instead it disappeared into a one-line changelog entry. The synthesis quality gate remains uncalibrated against that finding. Before adding the response processor or expanding synthesis capabilities, the observation-week findings should be written up formally.

- **The response processor described in Decision 026 overlaps with synthesis-service Class 1 in a way that isn't resolved.** The response processor detects "architectural decisions or observations in model output" and emits CaptureSuggestion candidates. Synthesis-service Class 1 consumes captures and emits ReviewItems. The response processor is upstream of captures (it's on the output side, detecting signals before they're explicitly captured). This is arguably distinct — output side detection vs. input capture processing. But the boundary is implicit. Without a formal boundary, the response processor will either duplicate synthesis-service logic or depend on it in ways that violate the single-responsibility principle.

- **`ActivationType.WORKFLOW` is a collision waiting to happen.** `workflow/` module, `/workflow/*` endpoints, `workflow.md` — "workflow" in this codebase means operational session/capture/review state management. `ActivationType.WORKFLOW` means "trigger a defined execution sequence." Different things. The reserved type costs nothing to rename now: `SEQUENCE`, `PROCEDURE`, or `EXECUTION_PLAN` all avoid the collision. This is a one-character enum value change while the code hasn't been written.

- **The pipeline assumes the pipeline is called on every prompt. That assumption needs a cost analysis before it ships.** The hook calls `POST /pipeline/process` on every UserPromptSubmit. The two-stage router (keyword filter → embedding comparison) keeps cost low for the common case. But if context-server is unreachable or slow, every prompt in a session acquires a network timeout. The hook "never-blocking" invariant from Decision 016 needs to be explicitly extended to the pipeline call: what is the timeout budget? What does graceful degradation look like when context-server doesn't respond in time?

- **pgvector HNSW indexes are a one-way door for the embedding model.** ChromaDB stored raw embeddings and could rebuild silently. PostgreSQL HNSW indexes are static until manually rebuilt with the correct vector dimension. Proposed-idea 003 (embedding model upgrade) was written pre-pgvector. An embedding model upgrade now requires: drop indexes, re-embed all 1249 docs, rebuild HNSW, re-index code symbols. The runbook doesn't exist. The proposed-idea doesn't acknowledge this cost. Before proposed-idea 003 advances, this migration cost should be documented.

- **The collaboration_runtime authority boundary conversation is the most important architectural discussion not yet had.** What can Shrey or Kyle do via Prismo's intelligence layer? Can they trigger synthesis runs? Create captures? Approve ReviewItems? Write proposed-ideas directly? The coding_runtime (Ethan via Claude Code) has implicit full authority. The collaboration_runtime needs explicit authority boundaries before the surface is built, because retrofitting auth after the interface is shipped is the pattern that produced V1's coupling problems. This is the "name the role before implementation" principle applied to a role that will have real authority implications.

- **Three Facets are listed in v2-progress remaining work (`/migrate`, `/sukuna`, `/review`) with no design record.** The architect Facet has a README, SKILL.md, facet.yaml, and a coherent reasoning posture. The three remaining Facets are names in a bullet list. Decision 026 established enough structure (context_loaders, heuristics, activation hints) that designing each Facet is a real design exercise. The sukuna Facet in particular is interesting: it would replace the sukuna.md agent definition and resolve the Decision 021 violation. It should be designed before being built.

- **The Service Rule has been enforcing "every route documented" since Phase 2.5. But the converse — "every documented capability has a contract block above the table" — has never been enforced.** The HTTP table has grown to 50+ entries. The contract definitions above (retrieve, synthesize, etc.) haven't kept pace. Pipeline capabilities are table-only. This asymmetry will compound as Phase 4 and 5 capabilities ship. The Service Rule should be extended: the startup check could also verify that every non-trivial capability in the HTTP table has a corresponding contract block. Or doctrine-service could catch it.

- **The 2-provider completion criterion from Decision 024 has no enforcement.** The rule: every capability must have ≥2 provider implementations before it's "complete." Currently: `embed` has one (SentenceTransformers). `synthesize` has two (AnthropicSynthesisProvider, ClaudeCodeSynthesisProvider). `analyze` has one. The pipeline assembler will add new capability types. None of these are tracked against the completion criterion. This is behavioral compliance right now. Until doctrine-service can query the CapabilityRegistry and surface single-provider capabilities as structural findings, this rule is aspirational. The Pydantic frontmatter validation work (trade_off draft from 2026-05-21) is the prerequisite for making this enforceable.

- **The pipeline module's `GET /pipeline/activations` endpoint is in the HTTP table but undocumented elsewhere.** Decision 026 defines `POST /pipeline/process` and `POST /pipeline/activate` in detail. There's no mention of `GET /pipeline/activations` in the decision — what does it return? A list of registered Activations with their hints and types? The endpoint was added to the Service Contracts table but its contract (request/response shape) is nowhere in the canon. This is a forward-registered Service Rule entry with no backing contract definition.
